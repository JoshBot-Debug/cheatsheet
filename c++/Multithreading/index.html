Multithreading

info

Async, Promise, Future:

If you're coming here from javascript or python, you are probably familiar with these terms, if you're thinking coroutines, you're wrong. :(

In 2011 the C++ added std::async, std::future, std::promise to the language. They were added in the STL to help manage threads better. Yes, thread, as in multithreading.

In 2014-2015 python stole async (asyncio). Python is single threaded so async was a coroutine.

In 2015-2017 javascript stole the keyword "Promise", JS is single threaded so Promise was a coroutines. They later added async & await.

Hopefully you'll be able to get over how annoying it is that this naming problem exists but well, here we are. std::async, std::promise and std::future was introduced into C++ first and it is multithreading.

Amdahl's law​

The core insight of Amdahl's Law is simple: the overall speedup of a program is limited by its strictly serial component. No matter how many cores you throw at a problem, the parts of the code that cannot be run in parallel (like initialization, disk I/O, or thread synchronization) will eventually act as a hard bottleneck on performance.

The Formula​

Slatency(s)=1(1−p)+pnS_{\text{latency}}(s) = \frac{1}{(1 - p) + \frac{p}{n}}Slatency​(s)=(1−p)+np​1​

Where:

  • SlatencyS_{\text{latency}}Slatency​ is the theoretical speedup of the whole task.
  • ppp is the proportion of the execution time that benefits from the improvement (the parallelizable part, expressed as a fraction between 0 and 1).
  • 1−p1 - p1−p is the proportion that cannot be improved (the strictly serial part).
  • nnn is the speedup factor of the part that benefits from the improvement (e.g., the number of processor cores).

Imagine a program takes 10 hours to run sequentially:

  • 9 hours (90%, so p=0.90p = 0.90p=0.90) can be parallelized.
  • 1 hour (10%, so 1−p=0.101 - p = 0.101−p=0.10) must run sequentially (e.g., saving the final file).

If you deploy this on a machine with an infinite number of processors, the 9-hour portion effectively shrinks to 0. However, the 1-hour serial portion still takes exactly 1 hour.

Your total execution time goes from 10 hours to 1 hour, resulting in a maximum possible speedup of 10x—even if you used 1,000 cores.

Calculated with the Formula (for 4 Cores):​

If we plug 4 cores (s=4s = 4s=4) into the formula for the same program:

Slatency=1(1−0.90)+0.904=10.10+0.225=10.325≈3.08S_{\text{latency}} = \frac{1}{(1 - 0.90) + \frac{0.90}{4}} = \frac{1}{0.10 + 0.225} = \frac{1}{0.325} \approx 3.08Slatency​=(1−0.90)+40.90​1​=0.10+0.2251​=0.3251​≈3.08

With 4 cores, the system runs roughly 3.08 times faster.

Multithreading APIs​

  • std::execution: Standard Execution Policies / Parallel Algorithms
  • std::async & std::promise:
    • std::async: A high-level function template used to run a task asynchronously (in a new thread or deferred) which returns a std::future holding the eventual result.
    • std::promise: The "input" or provider side of an asynchronous channel. A thread uses a std::promise to set a value or an exception, which is then delivered to the associated std::future.
  • std::thread & std::jthread: std::jthread automatically joins when it's destroyed. RAII.

Each of the topics above deserve a seperate book, let alone page. I'm going to cover it in a few paragraphs and examples. In the future I may add a seperate page for each and cover more details.

std::execution Standard Execution Policies​

The Standard Execution Policies:

  • std::execution::seq: Sequential execution (normal).
  • std::execution::par: Parallel execution on multiple threads.
  • std::execution::par_unseq: Parallel and unsequenced. This allows both multi-threading and SIMD (vectorization). It requires your loop body to be completely free of interleaving dependencies (no mutexes allowed inside here!).
  • std::execution::unseq (C++20): Vectorized execution on a single thread.
#include <algorithm>
#include <execution>
#include <vector>

std::vector<int> vec = { /* 1 kazillion elements */ };

// Multi-threaded sort
std::sort(std::execution::par, vec.begin(), vec.end());

There are 100s of algorithms in the standard library that have been overloaded to accept an execution policy as the first argument.

std::async, std::future, & std::promise​

The thread-safe delivery pipe designed to pass a value (or an exception) from a background worker thread to a waiting thread.

  • std::promise: The input end of the pipe. The worker thread uses it to store the result.
  • std::future: The output end of the pipe. The main thread holds this to retrieve the result. Calling .get() blocks the thread until the data arrives.
  • std::async: A helper function that automatically creates the pipe, spawns a background thread, executes your function, and hands you the std::future output.

std::promise and std::future cannot be copied, only moved std::move.

std::future can only call .get() once, it moves the value out.

std::shared_future can be copied and moved. Multiple threads can call .wait() & .get(). .get() returns a const T&.

#include <iostream>
#include <future>
#include <thread>
#include <chrono>

int heavy_cpu_work(int x) {
  std::this_thread::sleep_for(std::chrono::milliseconds(500));
  return x * x;
}

int use_async() {
  // Launch heavy_cpu_work(12) asynchronously on a separate OS thread
  std::future<int> result = std::async(std::launch::async, heavy_cpu_work, 12);
  return result.get(); // blocks the thread until the result arrives
}

int use_promise() {
  // Create the promise (input) channel
  std::promise<int> promise;

  // Get the corresponding future (output) channel from the promise
  std::future<int> future = promise.get_future();

  // Good old worker thread, move promise into it.
  std::thread worker([](std::promise<int> p) {
    int result = heavy_cpu_work(12);
    p.set_value(result); // Shove data into the pipe
    // p.set_execption() // Can also set an exeception
  }, std::move(promise));

  int result = future.get(); // blocks the thread until the result arrives OR throws the exception that was set.

  // Bad example I guess, in your program this would happen else where perhaps? (it should otherwise your program is redundant like this example)
  worker.join(); // Always remember to join your threads, or use std::jthread :P

  return result;
}

int main() {

  int a = use_async();
  int b = use_promise();

  std::cout << "use_async: " << a << std::endl;
  std::cout << "use_promise: " << b << std::endl;

  return 0;
}

A more realistic example:

std::future is returned by std::async and if you std::promise you'll use std::future. std::future is key here. To retrieve the data you have a few ways:

  • future.get(): Blocks the thread until the data is ready
  • future.wait_for(std::chrono::milliseconds(0)): Waits for a while or returns the data if it's ready.
  • future.wait_until(/**an absolute point in time, 3:15 PM**/): Waits until an absolute point in time. Like 3:15 PM.
#include <iostream>
#include <future>
#include <thread>
#include <chrono>

int heavy_cpu_work(int x) {
  std::this_thread::sleep_for(std::chrono::milliseconds(1500));
  return x * x;
}

int main() {
  std::future<int> result = std::async(std::launch::async, heavy_cpu_work, 12);

  // A game loop perhaps?
  while (true) {
    std::future_status status = result.wait_for(std::chrono::milliseconds(0));

    int square = 0;

    if (status == std::future_status::ready)
      square = result.get();

    // Gotcha: If std::async chose lazy evaluation, it will never finish unless you call get() or wait().
    if (status == std::future_status::deferred)
      square = result.get();

    // wait_for timed out
    // if (status == std::future_status::timeout)

    if(square) {
      std::cout << "Future is ready: " << square << std::endl;
      break;
    }

    std::cout << "Update & Render..." << std::endl;
    std::this_thread::sleep_for(std::chrono::milliseconds(200));
  }

  return 0;
}

Why use std::packaged_task instead of std::async?​

While std::async handles both creating the task and spawning the thread automatically, std::packaged_task decouples the two. This gives you much finer control:

  1. Custom Thread Management: You can hand a std::packaged_task off to a thread pool, an explicit std::jthread, or a specific background event loop, while still retaining the clean std::future API to get the result.
  2. Delayed Execution: You can create the task on one thread, pass it to an object or a queue, and execute it much later simply by invoking task(args...).
#include <iostream>
#include <thread>
#include <future>
#include <cmath>
#include <chrono>

int compute_power(int base, int exp) {
  std::this_thread::sleep_for(std::chrono::milliseconds(200));
  std::cout << "Meanwhile compute_power() finished... \n";
  return std::pow(base, exp);
}

int main() {
  std::packaged_task<int(int, int)> task(compute_power);

  std::future<int> future = task.get_future();

  // packaged_task is move-only; it cannot be copied
  std::jthread worker(std::move(task), 2, 10);

  std::cout << "[Main] Worker thread is calculating 2^10...\n";

  std::cout << "[Main] Doing other logic here...\n";
  std::this_thread::sleep_for(std::chrono::milliseconds(500));
  std::cout << "[Main] Doing more logic here...\n";
  std::this_thread::sleep_for(std::chrono::milliseconds(500));

  int result = future.get();

  std::cout << "[Main] Result received from worker: " << result << "\n";
  
  return 0;
}

std::thread & std::jthread​

info

std::jthread is short for Joining Thread.

Why std::jthread over std::thread:

  • Automatic Joining: std::jthread automatically calls .join() in its destructor if it's still joinable.
  • Built-in Cooperative Interruption: std::jthread natively supports cooperative cancellation using std::stop_token and std::stop_source.
  • std::stop_callback: Registers a callback function for an associated std::stop_token object.

std::stop_token: Cooperative Cancellation​

You may find yourself using a thread to run a task that should be cancellable, like rendering an image or downloading something. Historically, you would use an std::atomic<bool> as a flag in a std::thread.

std::jthread automaically passes a std::stop_token if your worker function accepts one as the first argument.

#include <iostream>
#include <thread>
#include <chrono>

void worker(std::stop_token stoken) {
  while (!stoken.stop_requested()) {
    std::cout << "Working...\n";
    std::this_thread::sleep_for(std::chrono::milliseconds(200));
  }
  std::cout << "Worker stopped.\n";
}

int main() {
  std::jthread jt(worker);

  std::this_thread::sleep_for(std::chrono::milliseconds(400)); // imagine the thread runs for a bit

  std::cout << "User pressed \"cancel\"\n";
  jt.request_stop(); 

  return 0;
}

Automatic Interruption​

#include <iostream>
#include <thread>
#include <condition_variable>
#include <mutex>
#include <chrono>
#include <queue>

std::mutex mtx;
std::condition_variable_any cv;
std::queue<int> queue;

void worker(std::stop_token st) {
  while (!st.stop_requested()) {
    std::unique_lock<std::mutex> lock(mtx);
    
    bool success = cv.wait(lock, st, [] {
      return !queue.empty();
    });

    // If wait returned because of a stop request, exit gracefully
    if (!success || st.stop_requested()) {
      std::cout << "Worker stopped." << std::endl;
      return;
    }

    // Process data
    int value = queue.front(); queue.pop();
    lock.unlock();

    std::cout << "Working..." << value << "\n";
  }
}

int main() {
  std::jthread t(worker);

  // Feed some data
  {
    std::lock_guard<std::mutex> lock(mtx);
    queue.push(42);
    cv.notify_one(); // notify one thread
  }

  std::this_thread::sleep_for(std::chrono::milliseconds(500));

  // jthread's destructor calls t.request_stop() and t.join() automatically. You can stop it earlier too if the user cancelled the operation, by calling t.request_stop().
  std::cout << "jthread exiting" << std::endl;
}
info

You must use std::condition_variable_any for this. The standard std::condition_variable does not have overloads that accept a std::stop_token.

std::stop_callback & std::stop_source​

  • std::jthread can be moved, but it cannot be copied. How do you stop the thread from another thread?
  • std::stop_source: get_stop_source() returns an std::stop_source. You can copy & move it to other threads.
  • The stop_source.request_stop() function returns true only on the very first successful call that actually triggers the stop. It returns false on all subsequent calls.
#include <iostream>
#include <thread>
#include <chrono>

void worker(std::stop_token stoken) {
  while (!stoken.stop_requested()) {
    std::cout << "Working...\n";
    std::this_thread::sleep_for(std::chrono::milliseconds(200));
  }
  std::cout << "Worker stopped.\n";
}

int main() {
  std::jthread t(worker);

  std::this_thread::sleep_for(std::chrono::milliseconds(400)); // imagine the thread runs for a bit

  auto onStopThread = []() {
    std::cout << "Stop callback executed" << std::endl;
  };

  std::stop_callback callback(t.get_stop_token(), onStopThread);

  std::cout << "User pressed 'cancel'" << std::endl;

  std::stop_source stop_source = t.get_stop_source();

  stop_source.request_stop();
  // Or directly call t.request_stop()
  // t.request_stop(); 

  return 0;
}
